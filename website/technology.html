Technology

Storage decisions you can audit.

ArgosFS splits the system into a POSIX-facing FUSE frontend, a shard-oriented data plane, a copy-on-write metadata layer, and a conservative health autopilot.

FUSE frontend Permission + ACL layer Stripe transform Placement scorer Metadata journal Health autopilot

Layered for experimentation, not mystery.

The data plane keeps file bytes in compressed and optionally encrypted erasure-coded stripes. The control plane keeps health, capacity, placement, workload, and policy state inspectable through metadata, CLI output, and JSONL experiment logs.

Subsystems

The implementation model.

Each subsystem is designed so a researcher can inspect its inputs and outputs rather than treating the filesystem as a black box.

Data plane

Stripe lifecycle

File data is split into stripes, compressed when configured, encrypted when enabled, encoded into k+m shards, written to selected disks, and referenced from metadata.

Redundancy

Reed-Solomon layouts

Configurable data and parity counts allow layouts such as 4+2. Degraded reads reconstruct missing shards when enough remaining shards are available.

Placement

Weighted rendezvous hashing

Placement scores combine stable hashing with disk tier, configured weight, observed latency, remaining capacity, NUMA locality, and failure-domain diversity.

Capacity

Hard reservation checks

Placement excludes disks with insufficient free shard capacity and rolls back reservations when writes fail, avoiding metadata that points at impossible allocations.

Metadata

Copy-on-write commits

Metadata writes use journal snapshots, multiple metadata copies, hash validation, replay, and mismatch repair to survive interrupted updates.

Security

Authenticated encryption

Argon2id-derived keys and XChaCha20-Poly1305 protect stripe contents; invalid or missing keys cause read/write denial rather than silent corruption.

I/O modes

Safe advanced I/O

Buffered I/O remains the baseline. Direct I/O and io_uring are attempted only when safe, with buffered fallback for unsupported or unaligned cases.

Observability

Metrics and artifacts

Health JSON, autopilot explanations, Prometheus metrics, fsck reports, CSV timings, and manifests make experiments reproducible.

Control plane

Automation is gated by invariants.

The autopilot plans maintenance from health and workload signals, but it must preserve redundancy, capacity feasibility, metadata consistency, and bootability before it can mutate the volume.

Safety

Never optimize through an unsafe state

Actions are rejected when they would violate redundancy or capacity constraints.

Explainability

Dry-run before mutation

Planner output records selected actions and rejected alternatives.

Bounded work

Maintenance is incremental

Scrub, rebalance, drain, and self-heal work are budgeted to control interference.

Verification

After-action checks close the loop

Journal, fsck, and shard validation can downgrade automation when assumptions fail.

Inspectability

Ask the planner before it mutates anything.

argosfs autopilot ROOT --dry-run --explain --json
argosfs health ROOT --json
argosfs verify-journal ROOT
argosfs fsck ROOT --repair --remove-orphans
